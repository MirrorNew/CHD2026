PUCT相关LLM树搜索框架的新增文献与新颖性再评估
执行摘要
本次检索以你指定的优先站点为起点，并扩展到官方论文页、会议页面、ACL Anthology、OpenReview、GitHub 和基准项目页；结论很明确：到 2025–2026 年，LLM+树搜索领域的主问题已经不再是“是否使用 PUCT”，而是“如何替换或改造 PUCT 的节点选择、预算分配、反馈整合与搜索对象表示”。与 Google 的 An AI system to help scientists write expert-level empirical software 最接近、且对你“网络瓦解 heuristics 自动发现”最有直接借鉴价值的新增工作，主要是 MCTS-AHD、Clade-AHD、A2DEPT、CodeTree、RethinkMCTS、HeuriGym/BLADE；其中 Clade-AHD 和 ReSCALE 对 PUCT 的替代意义最强，因为它们直接把问题指向 不确定性建模 与 预算可扩展性。基于新增文献，我对 PUCT 的再评估是：作为搜索规则本身，独创性偏中低；作为“可评分科学软件”系统中的工程骨架，独创性中高；作为跨学科应用系统（文献检索+重组+代码搜索+自动评分）的系统贡献，则仍然很强。对你的课题而言，最值得升级的不是继续“沿用 PUCT”，而是把 评价函数设计、树上不确定性、分阶段搜索、跨图分布泛化验证 一起纳入新的框架设计。 [1]
优先网站初检结果
按你的要求，我先在 arXiv、Google、dblp、X 上做了首轮检索，再决定扩展方向。首轮结果已经足以说明研究脉络正在从“AlphaZero 式 PUCT 树搜索”转向“预算/不确定性/反馈结构感知”的新一代框架。需要特别说明的是：Google 这篇论文在论文中把方法称为 PUCT tree search，而其开源仓库后来将参考实现命名为 Flat UCB Tree Search, FUTS；仓库原路径 google-research/score 现已重定向为 google-research/era。这不是两套不同系统，而是同一系统在“论文表述”和“开源实现命名”上的演化，且仓库 README 明确说该实现对应论文的 Algorithm 1，并在实现节再次写成 “Flat UCB (PUCT)” 。 [2]
从这一步就可以做出一个与你课题直接相关的判断：如果目标是“用 LLM+搜索自动发现网络瓦解 heuristic”，最该优先引用的不是通用 ToT/ReAct，而是 AHD/算法发现/代码搜索三类工作中真正处理“可评分程序搜索”的论文。 对应到文献层级，建议把 ERA、MCTS-AHD、Clade-AHD、A2DEPT、CodeTree、RethinkMCTS、HeuriGym/BLADE 放在第一梯队，把 ToT、ReAct、LATS、TS-LLM、PG-TD 放在方法学背景层。 [7]

关键观察
最接近你课题、也最值得主引的三篇，其实不是最热的通用 reasoning 论文，而是 ERA、MCTS-AHD、Clade-AHD。原因很简单：你的任务是“给图、给评分器、自动发现 node-removal heuristic”，这正是 AHD + scorable task 的交叉区域；而 ToT、ReAct、LATS 更多是通用 reasoning/agent 背景。 [37]
从“算法替代 PUCT”的角度看，最重要的新增认识有两个。第一，单节点均值 + UCB/PUCT 在 LLM 场景未必稳健，因为预算盖得更大时不一定更好，ReSCALE 已经给出明确反例；第二，有限预算下应该显式模型化不确定性，Clade-AHD 用 clade-level posterior + Thompson sampling 给出了一个很强的替代方向。对于你的网络瓦解场景，这两点都高度相关，因为 evaluator 通常耗时较高，图间分布差异也容易让“早期高分样本”误导后续扩展。 [38]
从“代码搜索对象”角度看，CodeTree 与 RethinkMCTS 提供了另外一种重要提醒：搜索不一定要直接在最终 heuristic code 上进行，也可以在思路、子目标、修复动作上搜索。这对你尤其关键，因为你现在的真正创新点很可能不是“找到某个神奇公式”，而是把 指标构造 与 启发式构造 联动起来，让树搜索同时探索“删点指标”与“评价指标”的解释性配合。 [39]
对 PUCT 的新颖性重新评估
先给结论
如果把“PUCT”本身当作论文的核心算法创新点，我给出的评分是 中低。如果把 Google 这篇工作理解为“把 LLM、树搜索、可评分科学任务、文献 idea injection、方法重组与大规模执行评估整合成一个科学软件发现系统”，我给出的评分是 中高到高。换句话说：PUCT 不是这条工作线中最独特的部分；真正独特的是系统化任务定义与实证落地能力。 [40]
分项评分
对你论文写作的更精确表述建议
你在相关工作里不宜写成“Google 提出了新型 PUCT 算法”。更稳妥、也更符合新增文献事实的写法是：
Google ERA 将 AlphaZero 启发的 PUCT/FUTS 适配到 可评分 empirical software 搜索中，并通过文献想法注入、方法重组和大规模执行评估展示了强系统能力；但后续文献显示，PUCT 本身并不是这一方向的终点，围绕预算可扩展性、不确定性建模、分阶段思路搜索和整程序树搜索，已经出现多条比标准 PUCT 更适合 LLM 场景的替代路线。 [44]
对你的课题再往前一步说，真正应该被你宣称为“新颖性核心”的，也不应是“我们用了 PUCT/Tree Search”，而应是两点：一是把 network dismantling 重新定义为 evaluator-sensitive scorable task；二是把 evaluator 从 GCC/R 拓展到 cNBI 这种更细粒度的深碎片化指标。 这两点在当前新增文献中仍然是相对空白的。AHD 文献大量优化 heuristic，但很少把 evaluator 设计本身当作第一等公民；而 HeuriGym、BLADE 反过来恰好提醒我们，没有强 benchmark 与强评价器，搜索框架的结论会很脆弱。 [45]
研究空白与未来工作建议
下面这部分，我专门按“是否直接服务你的网络瓦解工作”来排序，而不是按通用 LLM 热度排序。
这些建议并不是空泛地“再加点 agent/再加点 tree search”，而是与新增文献形成一一对应：Clade-AHD / ReSCALE 对应“替代 PUCT”；CodeTree / RethinkMCTS 对应“阶段化搜索与负反馈修树”；HeuriGym / BLADE 对应“标准化 benchmark”；A2DEPT / AlphaEvolve 对应“从 heuristic 发现走向 solver/system 发现”；ERA 对应“文献想法注入与重组”。 [46]
参考与资源
下面只列最值得你在论文和项目中直接保留的入口。为保持可点击性，我用引文链接而不是裸 URL。
首先是 Google 主线。论文与官方 publication page 适合作为系统主引；博客适合作为系统定位与应用补充；仓库与交互网站适合作为“开源实现与结果浏览”证据。 [47]
其次是 最相关的 AHD/树搜索论文与代码。MCTS-AHD 是你最该优先主引的 AHD 树搜索文献；CodeTree 和 RethinkMCTS 是代码搜索侧最值得并列比较的两篇；Clade-AHD、ReSCALE 则是你在“为什么不能默认 PUCT 最优”这一段最需要的证据。 [48]
再往上位一层，A2DEPT、AHD Agent、AlphaEvolve 分别代表三种替代思路：整程序树搜索、agentic RL 搜索、演化式代码代理。若你后续把网络瓦解从“单公式 heuristic”升级到“完整 solver”，它们会变得比 ToT/ReAct 更重要。 [49]
最后，HeuriGym、BLADE 与算法设计综述 推荐作为“为什么需要新 benchmark/新 evaluator”的佐证。HeuriGym 给出 agentic heuristic benchmark 与 QYI；BLADE 提供标准化 AAD 评测框架；ACM Computing Surveys 的综述则适合放在 related work 的统领性段落。 [50]
开放问题与局限
本报告也有边界。第一，若干 2026 年工作仍是 arXiv 预印本，例如 Clade-AHD、ReSCALE、A2DEPT、AHD Agent、Novelty-based ToT；我已优先引用原始预印本，但它们的最终会议版本、实验补充和代码成熟度仍可能变化。 [51]
第二，并不是所有论文都有官方代码。本报告只把我能从原始论文页或明确 GitHub 仓库确认的实现标为“有”；对其余条目，我只保守写成“本轮未见官方代码”，而没有推断其不可复现。 [52]
第三，目前仍缺少一篇直接在“heuristic discovery for graph/network dismantling”上，把 PUCT、Thompson-style clade selection、Gumbel+Sequential Halving、agentic RL、整程序树搜索放在同一预算与同一 evaluator 下做严格对比的论文。这恰好意味着：如果你下一步把 cNBI/BPSD 与上述替代树策略结合，并做系统对比，你会非常有机会贡献一篇真正“不是换提示词，而是改搜索范式与评价范式”的工作。

评价维度 | 评分 | 理由
算法层面的独创性 | 中低 | PUCT 本身来自 AlphaZero 传统；ERA 的真正改动是把它改造成 flat/global node selection、rank-score normalization、无 rollout 的 empirical-code search。这是一种有效适配，但不是从零提出一种新的树策略。后续工作如 ReSCALE、Clade-AHD、CMCTS 已直接给出更强的替代机制。 [41]
理论层面的独创性 | 低 | ERA 强在工程而非理论。论文没有给出树策略的 regret、sample-efficiency 或 scaling 分析；而 ReSCALE 至少明确识别出了 AlphaZero-style/PUCT 的 scaling failure，CMCTS 也从 action-space 约束角度解释了为什么常规 MCTS 在长链推理里会失效。 [42]
实证层面的独创性 | 高 | ERA 的实证覆盖 6 类科学问题，且不是单一 benchmark；Google Research 页面与博客都强调其在 scRNA-seq、COVID forecasting、时间序列、神经活动预测等多任务上的 expert-level 结果。这个广度在同类工作里依然突出。 [29]
应用层面的独创性 | 高 | “scorable scientific tasks / empirical software” 这一定义非常强，把程序发现问题转为了可自动评分的闭环科学工作流。对你这种网络瓦解任务尤为契合。 [29]
开源与可复用性 | 中高 | ERA 已开放论文、交互网站、最佳候选结果与参考实现；但完整内部实验环境、模型接入、计算基础设施与全部任务流水线仍不能完全由仓库复现。 [43]
